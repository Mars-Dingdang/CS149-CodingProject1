# Bonus Question: 10,000 Tasks vs. 10,000 Threads
这是一道非常经典的并行计算概念题。在 CS149 中，理解**线程 (Thread)** 与 **ISPC 任务 (Task)** 的区别，本质上是在理解“操作系统级并发”与“轻量级应用级调度”的区别。

以下是针对这 10,000 个执行单元在通用情况下的深度对比分析：

---

### 1. 资源开销与创建成本

* **10,000 个线程 (std::thread / pthreads):**
    * **重量级：** 每个线程都是操作系统的内核对象。在 Windows 或 Linux 上，创建一个线程需要分配独立的栈空间（通常为 1MB 或更多）和内核数据结构。
    * **后果：** 启动 10,000 个线程会导致内存压力剧增（仅栈空间就可能占用 10GB 内存），并且频繁的系统调用会让 CPU 忙于创建销毁，而不是执行计算。
* **10,000 个 ISPC Tasks:**
    * **轻量级：** ISPC Task 仅仅是一个数据结构（描述了函数指针和参数范围）。它不对应内核线程。
    * **后果：** 启动 10,000 个任务几乎是瞬间完成的，内存开销极小。



---

### 2. 调度机制：抢占式 vs. 协作式 (Worker Pool)

* **线程 (Thread):**
    * **抢占式调度：** 操作系统内核负责调度。当线程数远超 CPU 核心数（你的 ThinkBook 16p 可能只有 14-20 个逻辑核心）时，内核必须进行**上下文切换 (Context Switch)**。
    * **代价：** 频繁切换会导致缓存污染（Cache Misses）和寄存器保存/恢复开销，性能会大幅下降。
* **ISPC Task:**
    * **任务池模型 (Task Pool)：** ISPC 运行库通常维护一个固定数量的 Worker Threads（通常等于 CPU 核心数）。这 10,000 个 Task 被放进队列中。
    * **负载均衡：** 只有当一个核心完成了当前的 Task，它才会去队列里领取下一个。这种**协作式调度**避免了上下文切换，保持了高缓存局部性。

---

### 3. 语义与抽象目标

| 特性 | 线程 (Thread / Program 1) | ISPC Task (launch/sync) |
| :--- | :--- | :--- |
| **控制力** | 程序员手动控制任务分配（如 `start_row` 到 `end_row`） | 程序员描述**并行的可能性**，运行时负责分配 |
| **负载不均** | 如果某些线程任务重，其他核心会闲置（Wait） | **工作窃取 (Work Stealing)** 机制让闲置核心主动分担任务 |
| **同步** | `join()` 必须等待特定线程结束 | `sync` 等待当前函数发起的所有任务完成 |

---

### 4. 总结：10,000 个单元的实验结果

* **如果你启动 10,000 个线程：** 你的笔记本可能会出现明显的卡顿，系统忙于处理成千上万的时间片轮转，执行效率极低，甚至可能因为内存溢出或达到系统线程上限而报错。
* **如果你启动 10,000 个 ISPC Tasks：** 这是 ISPC 的**推荐用法**。它能极好地掩盖任务粒度不均的问题（比如 Mandelbrot 中某些区域计算快，某些慢）。ISPC 会高效地将这些小任务塞满你 CPU 的每一个物理核心。



---

### 💡 核心结论
**线程是关于“谁在干活”（资源），而 ISPC Task 是关于“活可以怎么分”（逻辑）。** 在并行计算中，我们倾向于使用大量的 Task 来“填满”少量的 Thread。
